Módulo cursos-service

Guía de trabajo para construir el servicio de cursos siguiendo el patrón del curso.

Propósito del módulo

Este servicio busca demostrar que el estudiante puede repetir el patrón técnico del curso con una entidad diferente. El valor pedagógico de cursos-service está en aplicar la misma arquitectura, pero sin depender de copiar sin entender.

Meta de aprendizaje: construir un segundo CRUD completo manteniendo consistencia técnica y seguridad.

Qué debe lograr el estudiante

  • Registrar cursos con código único.
  • Consultar y listar la oferta académica.
  • Actualizar y eliminar cursos.
  • Proteger los endpoints con token y roles.

Estructura sugerida

cursos-service/
├── client/AuthClient.java
├── controller/CursoController.java
├── handler/CursoHandler.java
├── service/CursoService.java
├── service/CursoServiceImpl.java
├── repository/CursoRepository.java
├── model/Curso.java
├── dto/CursoDTO.java
├── dto/CursoCreateDTO.java
└── security/AuthValidationFilter.java

Pasos guiados

1. Cree el proyecto base con la misma estructura de los otros servicios.
2. Diseñe la entidad Curso con codigo, nombre, descripcion, creditos y docenteResponsable.
3. Cree los DTO de entrada y salida.
4. Implemente repository, service, handler y controller.
5. Valide el token con auth-service.
6. Permita consulta a todos los roles definidos y escritura solo a ADMIN.
7. Verifique el servicio en Swagger.

Reglas mínimas de negocio

  • No registrar dos cursos con el mismo código.
  • Validar créditos mayores que cero.
  • Retornar mensaje claro cuando el curso no exista.
  • Proteger escritura con rol ADMIN.

Endpoints mínimos

EndpointMétodoRol esperado
/api/cursosGETADMIN, DOCENTE, ESTUDIANTE
/api/cursos/{id}GETADMIN, DOCENTE, ESTUDIANTE
/api/cursosPOSTADMIN
/api/cursos/{id}PUTADMIN
/api/cursos/{id}DELETEADMIN

Evidencias esperadas

  1. Listado de cursos.
  2. Creación exitosa de un curso.
  3. Intento fallido de escritura con rol no autorizado.
  4. Consulta o actualización de un curso específico.

Errores comunes

  • No validar la duplicidad del código.
  • Permitir modificaciones a cualquier rol.
  • Copiar el servicio anterior sin adaptar nombres ni reglas.
  • Dejar la documentación incompleta.

Checklist del módulo

  • CRUD completo de cursos.
  • Token obligatorio y roles correctos.
  • Regla de código único implementada.
  • Swagger funcional.
  • Evidencias listas.
