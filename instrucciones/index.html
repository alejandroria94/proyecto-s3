1. Introducción

El desarrollo de un sistema de software requiere una fase previa de análisis que permita comprender con claridad el problema, delimitar el alcance de la solución y definir la estructura general sobre la cual se construirá el proyecto. En este caso, el proyecto final del curso se enfoca en el diseño de una plataforma académica segura basada en microservicios, orientada a la administración de estudiantes, cursos y matrículas.

El análisis presentado en este informe busca establecer una base sólida para la implementación posterior, permitiendo que el equipo identifique de manera anticipada las responsabilidades de cada módulo, los flujos de información, las restricciones de seguridad y las decisiones técnicas más relevantes.

2. Descripción general del proyecto

El proyecto consiste en la construcción de una solución distribuida compuesta por varios microservicios, cada uno especializado en una responsabilidad específica del dominio académico. La propuesta toma como punto de partida una base desarrollada previamente en el curso y la amplía hacia un escenario más realista, donde la autenticación, la autorización, la gestión de entidades principales y la comunicación entre servicios forman parte integral del sistema.

Nombre del proyecto

Plataforma Académica Segura basada en Microservicios.

Propósito general

Administrar estudiantes, cursos y matrículas mediante servicios independientes protegidos con JWT.

3. Problema identificado

En muchos entornos académicos pequeños o en escenarios de práctica, la gestión de información suele desarrollarse mediante aplicaciones monolíticas o procesos manuales que dificultan la escalabilidad, la seguridad y el mantenimiento. Estas limitaciones se hacen más evidentes cuando se requiere separar responsabilidades, controlar el acceso según roles y exponer funcionalidades de forma organizada mediante APIs.

A partir de esta situación, se identifica la necesidad de una solución que permita:

  • centralizar la autenticación y la validación de usuarios;
  • separar la lógica de negocio en módulos independientes;
  • proteger los recursos del sistema mediante tokens;
  • facilitar la evolución del sistema sin afectar todos los componentes al mismo tiempo;
  • disponer de una estructura coherente para pruebas, documentación y despliegue.

4. Justificación

La construcción de este proyecto resulta pertinente porque permite aplicar, de forma integrada, los principales conceptos abordados durante el curso: arquitectura por capas, servicios REST, seguridad con JWT, documentación con Swagger, comunicación entre microservicios y despliegue reproducible mediante Docker.

Además de su valor técnico, el proyecto tiene un valor formativo importante, ya que obliga al estudiante a pasar de ejercicios aislados a una solución articulada, donde las decisiones de análisis impactan directamente la calidad de la implementación final.

5. Objetivos

5.1 Objetivo general

Analizar y estructurar una solución de software basada en microservicios para la gestión académica, definiendo sus componentes, requerimientos, restricciones y criterios técnicos antes de su implementación.

5.2 Objetivos específicos

  • Identificar el problema y las necesidades principales del sistema.
  • Definir los actores, el alcance y los procesos esenciales del proyecto.
  • Establecer los requisitos funcionales y no funcionales de la solución.
  • Describir la arquitectura propuesta y la función de cada microservicio.
  • Reconocer riesgos técnicos y criterios de validación del sistema.

6. Alcance del proyecto

El alcance del proyecto cubre el análisis y la posterior implementación de una plataforma académica básica con cuatro componentes principales:

auth-service

Gestiona el inicio de sesión, la generación de tokens JWT y la validación de credenciales.

estudiantes-service

Administra la información de los estudiantes y expone operaciones CRUD protegidas.

cursos-service

Administra la oferta académica, los cursos disponibles y sus datos principales.

matriculas-service

Registra y consulta matrículas, validando previamente la existencia de estudiantes y cursos.

Fuera de alcance: no se contempla en esta etapa la integración con bases de datos externas distribuidas, interfaz gráfica completa para usuarios finales, balanceadores, API Gateway ni mecanismos avanzados de observabilidad.

7. Actores del sistema

Actor Descripción Principales acciones
Administrador Usuario con privilegios completos sobre el sistema. Gestionar estudiantes, cursos, matrículas y consultar servicios protegidos.
Docente Usuario con permisos de consulta sobre parte del sistema. Consultar estudiantes, cursos y matrículas según las reglas definidas.
Estudiante Usuario con acceso restringido a información asociada a su rol. Consultar cursos y revisar sus propias matrículas.
Sistema de autenticación Componente que valida credenciales y firma tokens. Emitir y validar JWT.

8. Requisitos funcionales

  1. El sistema debe permitir el inicio de sesión mediante usuario y contraseña.
  2. El sistema debe generar un token JWT cuando las credenciales sean válidas.
  3. El sistema debe validar la autenticidad del token en los servicios protegidos.
  4. El sistema debe permitir registrar, consultar, actualizar y eliminar estudiantes.
  5. El sistema debe permitir registrar, consultar, actualizar y eliminar cursos.
  6. El sistema debe permitir registrar y consultar matrículas.
  7. El sistema debe verificar que un estudiante exista antes de matricularlo.
  8. El sistema debe verificar que un curso exista antes de registrar una matrícula.
  9. El acceso a los endpoints debe depender del rol del usuario autenticado.
  10. El sistema debe documentar sus endpoints mediante Swagger/OpenAPI.

9. Requisitos no funcionales

Categoría Requisito
Seguridad Los servicios protegidos deben exigir token Bearer válido.
Mantenibilidad El proyecto debe conservar una arquitectura organizada por capas.
Despliegue La solución debe poder ejecutarse con Docker Compose.
Usabilidad técnica Las APIs deben estar documentadas y probables desde Swagger UI.
Portabilidad El sistema debe poder ejecutarse localmente en distintos equipos del curso.
Consistencia Los servicios deben manejar respuestas estructuradas y errores controlados.

10. Reglas de negocio

  • Solo un usuario autenticado puede consumir servicios protegidos.
  • Solo el rol ADMIN puede crear, actualizar o eliminar estudiantes y cursos.
  • El rol DOCENTE puede consultar información, pero no modificar registros administrativos.
  • El rol ESTUDIANTE tiene acceso limitado a consultas específicas.
  • No se debe registrar una matrícula si el estudiante no existe.
  • No se debe registrar una matrícula si el curso no existe.
  • No se debe permitir una matrícula activa duplicada para el mismo estudiante y curso.

11. Análisis arquitectónico

La solución adopta una arquitectura basada en microservicios. Esta elección responde a la necesidad de separar responsabilidades, facilitar el mantenimiento y permitir que cada componente evolucione con menor acoplamiento.

11.1 Estilo arquitectónico

Cliente
   |
   | HTTP + Bearer Token
   v
Microservicios REST
   |
   +-- auth-service
   +-- estudiantes-service
   +-- cursos-service
   +-- matriculas-service

11.2 Patrón interno de cada servicio

controller -> handler -> service -> repository -> model
Interpretación del análisis: la arquitectura seleccionada no solo responde a una necesidad técnica, sino también didáctica, ya que permite evidenciar claramente la separación de responsabilidades y el flujo de cada solicitud.

12. Análisis por módulos

12.1 auth-service

Es el componente responsable de la autenticación. Su función es recibir credenciales, verificar al usuario, generar el token JWT y exponer un endpoint de validación utilizado por los demás servicios.

12.2 estudiantes-service

Se encarga de la administración de estudiantes. Este servicio representa una de las entidades principales del dominio y expone operaciones CRUD con validaciones básicas y control de acceso por rol.

12.3 cursos-service

Administra la información de los cursos y complementa el dominio académico. Su existencia permite separar la lógica de oferta académica respecto de la gestión de estudiantes.

12.4 matriculas-service

Orquesta la relación entre estudiantes y cursos. Su análisis es especialmente importante porque involucra validaciones cruzadas y comunicación entre microservicios, por lo cual introduce dependencias controladas mediante clientes Feign.

14. Riesgos y consideraciones técnicas

Riesgo de seguridad

Una configuración incorrecta del filtro JWT puede dejar recursos expuestos o bloquear accesos válidos.

Riesgo de integración

La comunicación entre servicios puede fallar si no se definen adecuadamente las URLs, puertos o contratos.

Riesgo de consistencia

Errores en validaciones cruzadas podrían permitir registros inconsistentes en matrículas.

Riesgo de despliegue

Una configuración incompleta en Docker Compose puede impedir la ejecución integrada del proyecto.

Consideración importante: aunque el sistema usa H2 para simplificar el entorno del curso, esta decisión implica que los datos son adecuados para prácticas y pruebas, no para persistencia productiva real.

15. Estrategia de validación

El análisis del proyecto debe complementarse con una estrategia de validación que permita comprobar el cumplimiento funcional y técnico de la solución. Para ello, se propone validar al menos los siguientes escenarios:

  • inicio de sesión exitoso y fallido;
  • acceso con token válido y acceso sin token;
  • consumo de endpoints según rol autorizado y no autorizado;
  • creación y consulta de estudiantes;
  • creación y consulta de cursos;
  • registro válido e inválido de matrículas;
  • levantamiento integral del sistema con Docker Compose.
Resultado esperado de la validación: el sistema debe demostrar coherencia entre el análisis realizado, la arquitectura propuesta y el comportamiento observado en las pruebas funcionales.

16. Conclusiones del análisis

El análisis realizado permite concluir que el proyecto propuesto es adecuado para integrar los conocimientos del curso en una única solución coherente. El problema identificado exige control de acceso, separación de responsabilidades y una arquitectura capaz de sostener crecimiento funcional sin perder claridad estructural.

Desde la perspectiva académica, este proyecto representa una oportunidad valiosa para que el estudiante aplique principios de análisis, modelado, diseño y documentación, antes de concentrarse en la implementación. La fase de análisis no debe entenderse como un trámite, sino como una herramienta que reduce errores posteriores y mejora la calidad del software desarrollado.

En consecuencia, este informe constituye la base conceptual para la construcción del proyecto final y orienta las decisiones técnicas que se ejecutarán en las siguientes fases del curso.
