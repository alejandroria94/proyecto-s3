Docker, Swagger y Pruebas

Guía final de empaquetado, orquestación, validación funcional y evidencias de entrega.

Propósito de esta etapa

Un proyecto no se considera completo solo porque funciona dentro del IDE. En esta fase el estudiante debe demostrar que puede documentar la solución, levantarla de forma reproducible y probarla como un sistema integrado.

Qué debe lograr el estudiante

  • Generar el .jar de cada servicio.
  • Crear un Dockerfile por microservicio.
  • Construir un docker-compose.yml que levante todo el sistema.
  • Validar endpoints y roles desde Swagger.
  • Reunir capturas o evidencias claras.
Paso obligatorio antes de Docker: los Dockerfile de este proyecto copian target/*.jar. Por eso, primero debe ejecutar mvn clean package -DskipTests dentro de cada servicio y luego sí usar docker compose up --build.

Dockerfile sugerido por servicio

FROM eclipse-temurin:21-jdk
WORKDIR /app
COPY target/*.jar app.jar
EXPOSE 8081
ENTRYPOINT ["java","-jar","/app/app.jar"]

El puerto debe ajustarse según cada servicio. Por ejemplo: 8081 para auth, 8082 para estudiantes, 8083 para cursos y 8084 para matrículas.

docker-compose.yml sugerido

services:
  auth-service:
    build: ./auth-service
    ports:
      - "8081:8081"

  estudiantes-service:
    build: ./estudiantes-service
    ports:
      - "8082:8082"
    depends_on:
      - auth-service

  cursos-service:
    build: ./cursos-service
    ports:
      - "8083:8083"
    depends_on:
      - auth-service

  matriculas-service:
    build: ./matriculas-service
    ports:
      - "8084:8084"
    depends_on:
      - auth-service
      - estudiantes-service
      - cursos-service

Pruebas mínimas en Swagger

  • Login exitoso con admin.
  • Login fallido con contraseña incorrecta.
  • Acceso rechazado sin token.
  • Acceso exitoso con token válido.
  • Creación de estudiante y curso con rol ADMIN.
  • Intento fallido de escritura con rol no autorizado.
  • Registro de matrícula válido.
  • Intentos inválidos de matrícula por inexistencia o duplicidad.

Comandos sugeridos

mvn clean package
docker compose up --build

Antes de usar Docker Compose, asegúrese de que cada servicio compila correctamente por separado.

Evidencias que debe entregar el estudiante

  1. Captura de docker compose up mostrando los servicios en ejecución.
  2. Captura de Swagger en cada servicio.
  3. Capturas de pruebas exitosas y fallidas.
  4. Archivo docker-compose.yml.
  5. Dockerfile de cada microservicio.

Errores comunes

  • No compilar los .jar antes de construir la imagen.
  • Exponer puertos incorrectos.
  • No ajustar las URL internas cuando se usa Docker.
  • Entregar capturas sin contexto o sin mostrar el endpoint probado.

Checklist final de la etapa

  • Todos los servicios compilan.
  • Docker Compose levanta el sistema.
  • Swagger funciona en cada servicio.
  • Las pruebas mínimas fueron ejecutadas.
  • Las evidencias son claras y suficientes.

Apoyo conceptual

Antes de ejecutar las pruebas de seguridad, revise la guía específica del sitio para comprender qué significa autenticar, autorizar, validar el token y restringir un endpoint por rol.

Respuestas de seguridad uniformes

Los servicios protegidos ahora responden con una estructura JSON uniforme cuando ocurre un error de autenticación o autorización. Esto mejora la trazabilidad, facilita las pruebas en Swagger y permite que el cliente identifique con claridad si el problema fue un token ausente, un token inválido o un rol insuficiente.

{
  "success": false,
  "message": "Debe enviar un token Bearer válido",
  "errorCode": "AUTH_HEADER_MISSING",
  "status": 401,
  "path": "/api/recurso",
  "timestamp": "2026-04-11T15:30:00"
}
  • AUTH_HEADER_MISSING: no se envió el encabezado Bearer.
  • TOKEN_INVALID: el token no es válido o no contiene rol.
  • TOKEN_VALIDATION_ERROR: no fue posible validar el token contra auth-service.
  • AUTH_FORBIDDEN: el usuario está autenticado, pero no tiene el rol necesario.
