Propósito del proyecto

El proyecto final tiene como objetivo consolidar los aprendizajes del curso en una sola solución. A diferencia de los ejercicios aislados, aquí el estudiante debe demostrar que comprende cómo se articulan autenticación, seguridad, arquitectura por capas, comunicación entre servicios, documentación y despliegue.

La propuesta se apoya en el laboratorio base del curso. Por eso no se trata de construir algo totalmente distinto, sino de ampliar una base conocida hasta convertirla en una plataforma académica sencilla, segura y distribuida.

Flujo recomendado para usar el código entregado: 1) compile cada microservicio con Maven, 2) verifique login y endpoints en Swagger, 3) levante el conjunto completo con Docker Compose, y 4) registre las evidencias pedidas en la guía.

Caso propuesto

Una institución académica requiere una plataforma para administrar estudiantes, cursos y matrículas. Para evitar mezclar toda la lógica en una sola aplicación, la solución se divide en cuatro servicios: autenticación, estudiantes, cursos y matrículas. El sistema debe permitir iniciar sesión, obtener un token y consumir operaciones protegidas de acuerdo con el rol del usuario.

Competencias a demostrar

Técnicas

  • Construir APIs REST con Spring Boot.
  • Organizar el código en controller, handler, service, repository y model.
  • Aplicar BCrypt y JWT.
  • Consumir otros servicios mediante OpenFeign.
  • Documentar con Swagger.
  • Ejecutar con Docker Compose.

Formativas

  • Justificar decisiones arquitectónicas.
  • Construir por fases y no de forma improvisada.
  • Entregar una solución verificable y bien explicada.
  • Probar escenarios exitosos y fallidos.

Importante: primero se debe cumplir el alcance mínimo. Las mejoras adicionales solo tienen sentido si la base ya funciona.

Arquitectura esperada

Cada microservicio debe mantener la estructura trabajada en clase:

controller → handler → service → repository → model

Esto permite mantener controladores livianos, una capa de negocio clara y persistencia desacoplada. Además, todos los servicios deben exponer sus propios endpoints, configuración y documentación Swagger.

Secuencia guiada de desarrollo

Paso 1. Revise el proyecto base y verifique que el laboratorio anterior funcione.
Paso 2. Cree la estructura general del repositorio con las carpetas de los cuatro servicios.
Paso 3. Implemente primero auth-service.
Paso 4. Implemente estudiantes-service reutilizando el CRUD base.
Paso 5. Implemente cursos-service siguiendo el mismo patrón.
Paso 6. Implemente matriculas-service cuando los servicios de estudiantes y cursos ya funcionen.
Paso 7. Cree Dockerfiles, documente con Swagger y pruebe el sistema completo.

Guía de apoyo sobre seguridad

Como complemento de esta guía general, el sitio incluye una página específica de seguridad donde se explica el flujo de autenticación, el uso de JWT, la diferencia entre autenticación y autorización, la validación del rol y las pruebas mínimas que deben realizarse para sustentar el proyecto.

Checklist final

  • El login funciona y genera token.
  • Los servicios protegidos rechazan peticiones sin token.
  • Los roles limitan correctamente el acceso.
  • Estudiantes y cursos tienen CRUD completo.
  • Las matrículas validan existencia y duplicidad.
  • Swagger permite probar todo el sistema.
  • Docker Compose levanta todos los servicios.

Respuestas de seguridad uniformes

Los servicios protegidos ahora responden con una estructura JSON uniforme cuando ocurre un error de autenticación o autorización. Esto mejora la trazabilidad, facilita las pruebas en Swagger y permite que el cliente identifique con claridad si el problema fue un token ausente, un token inválido o un rol insuficiente.

{
  "success": false,
  "message": "Debe enviar un token Bearer válido",
  "errorCode": "AUTH_HEADER_MISSING",
  "status": 401,
  "path": "/api/recurso",
  "timestamp": "2026-04-11T15:30:00"
}
  • AUTH_HEADER_MISSING: no se envió el encabezado Bearer.
  • TOKEN_INVALID: el token no es válido o no contiene rol.
  • TOKEN_VALIDATION_ERROR: no fue posible validar el token contra auth-service.
  • AUTH_FORBIDDEN: el usuario está autenticado, pero no tiene el rol necesario.
