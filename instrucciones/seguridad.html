Propósito

La seguridad del proyecto busca garantizar que cada solicitud sea ejecutada solo por usuarios autenticados y con permisos suficientes. En este sistema, autenticación y autorización no son lo mismo: primero se verifica la identidad y después se decide qué puede hacer el usuario.

Idea central: tener un token válido no significa tener permiso para todo. El rol define el alcance del acceso.

Conceptos clave

Autenticación

Proceso de verificar si el usuario realmente es quien dice ser. Aquí ocurre en /auth/login.

Autorización

Proceso de decidir qué operaciones están permitidas según el rol del usuario autenticado.

JWT

Token firmado que viaja en el encabezado Authorization y evita reenviar credenciales en cada petición.

BCrypt

Algoritmo usado para almacenar contraseñas cifradas en lugar de texto plano.

Arquitectura de seguridad

  • auth-service recibe credenciales y genera el JWT.
  • Los servicios de negocio reciben el token como Bearer Token.
  • Un filtro valida el token antes de ejecutar lógica de negocio.
  • SecurityConfig revisa el rol para autorizar o negar el acceso.
Usuario → auth-service → JWT → estudiantes/cursos/matriculas → validación del token → revisión del rol → respuesta

Rol de auth-service

Este servicio centraliza la autenticación. Busca el usuario, compara la contraseña con BCrypt, genera el token y ofrece un endpoint para validarlo. Así se evita repetir login en los demás microservicios.

  • /auth/login: recibe credenciales y retorna JWT.
  • /auth/validate: valida el token y retorna si es válido, el usuario y el rol.

JWT y contenido del token

El token debe contener información mínima y útil para los demás servicios.

  • sub: usuario autenticado.
  • role: rol del usuario.
  • fecha de emisión.
  • fecha de expiración.
{
  "sub": "admin",
  "role": "ADMIN",
  "iat": 1712800000,
  "exp": 1712803600
}

Flujo de autenticación

1. El usuario envía username y password a /auth/login.
2. El sistema busca el usuario y compara la contraseña con BCrypt.
3. Si todo es correcto, genera el JWT.
4. El cliente guarda el token y lo envía en las siguientes peticiones.
5. Cada servicio valida el token antes de atender la solicitud.
6. Spring Security revisa el rol antes de permitir el acceso al endpoint.

Roles y autorización

RolPuede hacerNo debería hacer
ADMINGestionar estudiantes, cursos y matrículas.—
DOCENTEConsultar estudiantes, cursos y matrículas.Modificar información administrativa.
ESTUDIANTEConsultar cursos y, si se implementa, sus propias matrículas.Administrar estudiantes o cursos.

Las validaciones de rol se aplican normalmente en SecurityConfig mediante reglas como hasAuthority("ADMIN") o hasAnyAuthority("ADMIN", "DOCENTE").

Filtros y contexto de seguridad

El filtro de autenticación toma el encabezado Authorization, valida el token y carga el usuario con su rol dentro del contexto de Spring Security. Gracias a eso, los controladores no tienen que validar manualmente el rol.

Authorization: Bearer eyJhbGciOiJIUzI1NiJ9...
Importante: la lógica de autorización no debe mezclarse con la lógica del negocio. Primero pasa por el filtro y por la configuración de seguridad.

Errores comunes

  • Guardar contraseñas sin cifrado.
  • No incluir el rol dentro del token.
  • Permitir acceso a endpoints protegidos sin token.
  • Asumir que un token válido da permisos totales.
  • No probar casos fallidos como token ausente, inválido o rol insuficiente.

Pruebas mínimas

  • Login exitoso con credenciales válidas.
  • Login fallido con contraseña incorrecta.
  • Acceso rechazado sin token.
  • Acceso exitoso con token válido.
  • Intento de operación con rol no autorizado.
  • Intento con token inválido o expirado.

Respuestas de seguridad uniformes

Los servicios protegidos ahora responden con una estructura JSON uniforme cuando ocurre un error de autenticación o autorización. Esto mejora la trazabilidad, facilita las pruebas en Swagger y permite que el cliente identifique con claridad si el problema fue un token ausente, un token inválido o un rol insuficiente.

{
  "success": false,
  "message": "Debe enviar un token Bearer válido",
  "errorCode": "AUTH_HEADER_MISSING",
  "status": 401,
  "path": "/api/recurso",
  "timestamp": "2026-04-11T15:30:00"
}
  • AUTH_HEADER_MISSING: no se envió el encabezado Bearer.
  • TOKEN_INVALID: el token no es válido o no contiene rol.
  • TOKEN_VALIDATION_ERROR: no fue posible validar el token contra auth-service.
  • AUTH_FORBIDDEN: el usuario está autenticado, pero no tiene el rol necesario.
