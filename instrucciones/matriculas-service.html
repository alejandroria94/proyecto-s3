Módulo matriculas-service

Guía de trabajo para el servicio integrador con reglas de negocio y consumo entre servicios.

Propósito del módulo

matriculas-service es el módulo más integrador del proyecto. Aquí el estudiante ya no resuelve un CRUD aislado, sino una operación que depende de otros servicios y que exige reglas de negocio más cuidadosas.

Meta de aprendizaje: comprender por qué la arquitectura distribuida tiene sentido cuando una operación necesita validar información en más de un servicio.
Refactor aplicado en este repositorio: MatriculaService ya no recibe el encabezado Authorization. El token se propaga automáticamente a Feign y los errores remotos ahora se reportan con más contexto.

Qué debe lograr el estudiante

  • Registrar matrículas con estudianteId y cursoId.
  • Consultar y listar matrículas.
  • Anular matrículas cambiando su estado.
  • Validar existencia de estudiante y curso antes de guardar.
  • Evitar matrículas activas duplicadas.

Secuencia de registro

1. Llega una solicitud con estudianteId y cursoId.
2. El servicio consulta estudiantes-service para verificar el estudiante.
3. El servicio consulta cursos-service para verificar el curso.
4. Revisa si ya existe una matrícula activa para la misma combinación.
5. Si todo es válido, guarda la matrícula con estado ACTIVA.

Reglas mínimas de negocio

  • No registrar una matrícula si el estudiante no existe.
  • No registrar una matrícula si el curso no existe.
  • No duplicar una matrícula activa para el mismo estudiante y curso.
  • La anulación debe cambiar el estado y no necesariamente eliminar físicamente el registro.

Evidencias esperadas

  1. Registro exitoso de una matrícula.
  2. Intento fallido porque el estudiante no existe.
  3. Intento fallido porque el curso no existe.
  4. Intento fallido por duplicidad.
  5. Anulación correcta de una matrícula.

Errores comunes

  • Registrar sin validar estudiante y curso.
  • No manejar errores de integración entre servicios.
  • Permitir duplicidad de matrículas activas.
  • Ubicar toda la lógica en el controlador.

Checklist del módulo

  • Se consumen correctamente estudiantes-service y cursos-service.
  • Existen reglas de negocio claras en la capa service.
  • El token es obligatorio.
  • Se probaron casos positivos y negativos.
  • Swagger documenta correctamente el módulo.

Respuestas de seguridad uniformes

Los servicios protegidos ahora responden con una estructura JSON uniforme cuando ocurre un error de autenticación o autorización. Esto mejora la trazabilidad, facilita las pruebas en Swagger y permite que el cliente identifique con claridad si el problema fue un token ausente, un token inválido o un rol insuficiente.

{
  "success": false,
  "message": "Debe enviar un token Bearer válido",
  "errorCode": "AUTH_HEADER_MISSING",
  "status": 401,
  "path": "/api/recurso",
  "timestamp": "2026-04-11T15:30:00"
}
  • AUTH_HEADER_MISSING: no se envió el encabezado Bearer.
  • TOKEN_INVALID: el token no es válido o no contiene rol.
  • TOKEN_VALIDATION_ERROR: no fue posible validar el token contra auth-service.
  • AUTH_FORBIDDEN: el usuario está autenticado, pero no tiene el rol necesario.
